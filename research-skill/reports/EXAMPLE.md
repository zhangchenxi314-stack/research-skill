## 大模型训练与微调
训练稳定性和微调效率是本周围绕大模型的两个核心技术话题。Loss spike 的根因分析有了系统性的分类框架，LoRA 微调的最佳参数配置通过消融实验达成了较明确的共识。长上下文位置编码的比较研究也为需要处理超长文档的场景提供了选型参考。

📌 落地案例
- [大模型训练稳定性：Loss Spike 根因分析与修复](https://3ms.huawei.com/article/20240625-03) — 3ms
  将训练过程中常见的 loss spike 归因为三类：数据异常（脏数据、重复样本）、优化器状态损坏（Adam 二阶动量溢出）、混合精度溢出（FP16 下梯度消失/爆炸）。文章给出了每种根因的检测脚本和针对性修复策略，并提供了一个训练监控 dashboard 的设计方案，建议接入训练平台的标准工具链。

- [LoRA 微调参数选择指南：rank、alpha、dropout](https://jxsq.huawei.com/topic/20240625-03) — jx社区
  通过消融实验系统对比了 rank (4/8/16/32/64)、alpha 和 dropout 在 7B-13B 模型上的表现。核心结论：rank=16 在大多数任务上达到性价最优，继续增大 rank 收益递减；dropout=0.05 能有效防止过拟合但对最终效果影响很小。文末附了一份「按场景选参数」的速查表。

- [大模型长上下文窗口的位置编码研究](https://2012lab.huawei.com/paper/20240625-02) — 2012实验室
  在 128K 上下文窗口下系统比较了 RoPE、ALiBi、NoPE 三种位置编码方案。RoPE 在短距离依赖上表现最好但长距离衰减明显，ALiBi 长距离外推能力强但短距离精度略差。论文提出了一种混合方案：短距离用 RoPE、长距离切换到 ALiBi，在保持短距离精度的同时将 128K 位置的困惑度降低了 12%。

## Agent 生产级实践
Agent 本周最大的亮点是三个完整的生产系统集中曝光——告警自愈、研发效能、网络运维。这些案例的共同特征是都采用了多 Agent 协作架构，且给出了明确的生产指标（MTTR、效率提升、自动化率）。Agent 正在从「有趣的技术」变成「可衡量的生产力工具」。

📌 落地案例
- [SRE 团队基于多 Agent 的告警自愈系统](https://3ms.huawei.com/article/20240625-05) — 3ms
  这是目前公司内部规模最大的多 Agent 生产系统之一。系统由三个专业 Agent 组成：告警分级 Agent（判断严重程度和影响面）、日志诊断 Agent（拉取关键日志并定位根因）、修复执行 Agent（匹配已知修复方案并执行回滚/扩容等操作）。三个 Agent 通过消息总线串行协作。已接入 200+ 微服务，当前自动修复成功率为 72%，MTTR 从 45 分钟降至 8 分钟。团队下一步计划引入辩论机制来提高诊断准确率。

- [Agent 模式在内部研发效能平台的应用](https://jxsq.huawei.com/topic/20240625-04) — jx社区
  基于 ReAct 模式实现了一条从需求分析到代码生成的自动流水线：Agent 读取 TAPD 需求 → 理解功能点 → 检索内部代码库中的相似实现 → 生成初版代码 → 自动运行单测 → 提交 MR。目前在试点团队的落地效果是开发效率提升约 30%，但代码质量仍需人工 review——Agent 倾向于生成能通过测试但架构不合理的代码，这是社区讨论的热点争议。

- [多 Agent 协作架构在电信网络运维中的实践](https://w3.huawei.com/page/20240625-01) — w3
  电信网络运维场景复杂、变更风险高，单 Agent 难以覆盖。该方案采用三个专业 Agent：配置管理 Agent（操作网元配置）、故障诊断 Agent（分析告警关联和根因）、性能优化 Agent（分析 KPI 和容量规划）。Agent 之间通过消息总线异步协作，每个 Agent 有独立的知识库和工具集。已在 5G 核心网试点，配置变更的自动化率从 15% 提升至 60%。

## Agent 工程化与安全
Agent 要规模化铺开，框架选型和安全护栏是两个必须回答的横向问题。框架选型方面，LangGraph 和 CrewAI 各有优劣，没有万能方案。安全方面，三层护栏架构（输入校验→工具权限→输出审核）已从设计走向落地。

📌 落地案例
- [Agent 框架选型：LangGraph vs CrewAI vs 自研](https://3ms.huawei.com/article/20240625-06) — 3ms
  从三个维度做了系统对比。任务编排方面，LangGraph 的图结构对复杂工作流支持最好但学习曲线陡峭，CrewAI 上手快但动态调整能力弱。状态管理方面，LangGraph 有内置的 checkpoint 机制，CrewAI 需要自己管理。可观测性方面，自研方案可以对接到内部的日志和监控体系，开源框架则需要额外适配。最终建议：简单串联任务用 CrewAI，复杂条件分支任务用 LangGraph，需要深度定制和内部系统对接的场景考虑自研。

- [Agent 安全护栏设计规范](https://w3.huawei.com/page/20240625-02) — w3
  提出 Agent 的三层安全架构：第一层输入校验——对 LLM 的输入做敏感词过滤和 prompt injection 检测；第二层工具调用权限控制——定义每个工具的最小权限集和调用频率上限，防止 Agent 越权操作或死循环调用；第三层输出审核——对 Agent 的输出做合规检查后才允许执行。该方案已在内部 Agent 平台落地，拦截了 0.3% 的异常调用。

## RAG 检索优化与知识库构建
RAG 的优化方向本周集中在三个层面：检索策略从关键词升级到混合检索、Chunking 策略从经验驱动转向实验驱动、知识库构建从零散操作走向全流程标准化。混合检索（BM25 + 向量）已基本成为标配，语义分块在长文档场景的优势得到多个团队验证。

📌 落地案例
- [客服知识库 RAG 系统升级：从关键词检索到混合检索](https://3ms.huawei.com/article/20240625-07) — 3ms
  原有的客服知识库 RAG 仅使用关键词检索，长尾问题召回率只有 67%。升级为 BM25 + 向量检索的混合方案（权重 3:7）后，Top-5 召回率提升到 89%。配合对答案的重排序和置信度过滤，客服机器人的首次解决率从 45% 提升到 60%。系统已全量上线，日均处理 50 万次问答。团队下一步计划引入多模态检索来支持截图类问题的定位。

- [RAG 系统中 Chunking 策略的对比实验](https://jxsq.huawei.com/topic/20240625-05) — jx社区
  在内部文档集上系统对比了三种分块策略：固定长度（512 token）简单但容易切断语义、语义分块（按段落/章节边界）在长文档场景的检索准确率比固定长度高 18%、递归分块（多粒度同时索引）效果最好但存储开销大 2-3 倍。社区的共识是：短文档用固定长度即可，长文档优先使用语义分块，对检索质量要求极高的场景考虑递归分块。

- [知识库构建方法论：从文档到向量库](https://w3.huawei.com/page/20240625-03) — w3
  系统总结了从原始文档到可检索向量库的全流程：文档解析（PDF/Word/Confluence 多格式）→ 清洗（去页眉页脚、表格规范化）→ 分块 → 嵌入模型选型（内部自研 vs 开源）→ 索引构建 → 质量评估。附带了一份质量评估 checklist，涵盖召回率、精确率、响应延迟等 12 个指标，可作为新项目启动时的模板直接复用。

## 协议标准化与安全前沿
MCP 协议和 AI 安全本周各自只有一篇内容，但方向互补——一个关注 Agent 如何安全地接入工具生态，一个关注大模型本身如何防御对抗攻击。两者都处于早期探索阶段，尚无实际的生产级部署案例，但方法论建设正在加速。

📌 落地案例
- [MCP 协议的扩展性分析：从单工具到工具生态](https://2012lab.huawei.com/paper/20240625-03) — 2012实验室
  从三个核心设计维度分析了 MCP 协议：工具发现机制通过注册中心 + 能力描述实现；权限模型通过 OAuth 2.0 + 细粒度 scope 实现；流式传输支持工具调用的实时进度反馈。论文还讨论了多 Agent 场景下的工具冲突和共享问题，提出了一种基于租约的分布式锁方案，但该部分目前只是设计提案，尚未有实现验证。
